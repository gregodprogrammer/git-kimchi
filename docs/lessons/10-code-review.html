Code Review

← Previous: Pull Requests and Issues · ← Back to Index

Why Code Review Matters

Code review is the practice of having another person examine your changes before they become part of the codebase. It is a cornerstone of healthy software teams, and in a DevOps culture it is not just a quality gate — it is a collaboration ritual.

Benefits for your team:

  • Catch bugs early — A second pair of eyes finds issues before they reach production. Fixing a bug in review costs a fraction of what it costs in production.
  • Share knowledge — Code review spreads understanding of the codebase across the team. No single person becomes a bottleneck.
  • Maintain standards — Over time, consistent review enforces coding style, architectural patterns, and documentation expectations.
  • Onboard new members faster — New engineers learn the codebase and team conventions by reading others' PRs and getting their own reviewed.

DevOps context: In a CI/CD pipeline, code review is the "human" stage before automation takes over. Your pipeline can run tests, but it cannot notice that a variable name is confusing, that error handling is missing, or that the new dependency introduces a security risk. That is the reviewer's job.

A note on trust: Code review is not about catching mistakes to blame people. It is about improving code together. The best teams treat it as a normal, expected part of the workflow — not as a confrontation.

The Review Mindset

How you approach a review matters as much as what you look for. A good reviewer and a good author both follow the same principle: focus on the code, not the person.

Giving Feedback

Do: Describe what the code does and why it might be a problem.

# Instead of:
"You forgot to handle the error case."

# Say:
"This function does not handle the case where the API returns a 403.
That could cause an unhandled exception for users whose tokens expire mid-session."

The second version describes the specific situation, explains the consequence, and gives the author enough context to understand and fix it without feeling attacked.

Avoid: Absolute language that sounds like a verdict. Words like "wrong," "stupid," "always," or "never" put people on the defensive and make conversations less productive.

Receiving Feedback

When someone leaves a review comment on your PR:

  1. Read it fully before responding. Do not assume you know what they mean.
  2. Ask clarifying questions if the comment is unclear. A genuine question is not a challenge — it is useful signal.
  3. Acknowledge the feedback, even if you disagree. A simple "Good point, I will fix it" or "I had considered that — here is why I chose this approach" opens a productive conversation.
  4. Do not take it personally. The reviewer is looking at your code, not at you.

Types of Feedback

Not every comment on a PR carries the same weight. Understanding the difference helps both reviewers and authors prioritize what to address first.

Blocking Feedback

Blocking comments indicate a problem that must be resolved before the PR can merge. These are typically:

  • Bugs or logic errors
  • Missing tests for core functionality
  • Security vulnerabilities (e.g., unsanitized user input, hardcoded secrets)
  • Breaking changes that were not discussed
  • Missing documentation for public APIs

Example blocking comment:

"This null check is missing. If `user` is `None` here, the line below will
raise an `AttributeError` at runtime. Please add a guard before proceeding."

Non-Blocking Feedback (Suggestions / Nits)

Non-blocking comments are ideas for improvement that do not prevent merging. They include:

  • Style preferences (indentation, naming conventions)
  • Simplifications that make code more readable
  • Suggestions for future improvements beyond the PR scope

Prefix suggestions with a marker so the author knows the urgency:

"Nit: this could be a one-liner with a list comprehension."

The term "nit" comes from "nitpick" — small, minor issues. If everything in a PR is a nit, the PR can merge. If anything is blocking, it should not.

Questions

Questions are not feedback at all — they are requests for understanding:

"Why do we need to retry here? Is the API known to be unreliable, or is this
protecting against a specific edge case I am not aware of?"

A question invites discussion. The author might realize the code needs to change, or they might explain a constraint the reviewer was missing. Either way, both people learn something.

GitHub Review Features

GitHub provides a rich interface for conducting reviews directly in the browser. Familiarize yourself with these tools — they make reviews faster and more precise.

PR Diff View

When you open a PR on GitHub, you see a split or unified diff. Lines that were added appear in green (with a + prefix in unified view), and lines removed appear in red (with a - prefix). The Files Changed tab shows every modified file.

At the top of the Files Changed view, you can switch between unified diff (all changes in one column) and split diff (old version on the left, new version on the right). Split view is often easier to read for large changes.

Line-Level Comments

Click the + button to the left of any line in the diff to add a comment directly to that line. This is the most precise way to leave feedback — the comment is anchored to the exact code in question.

When you add a line-level comment, GitHub gives you two options:

  • Add a single comment — Leaves a comment without formally starting a review.
  • Start a review — Holds your comments and submits them all together with a summary.

For anything more than a quick note, start a review and submit all comments at once. This reduces notification noise and gives the author a comprehensive picture of what needs attention.

Suggested Changes

For small, specific fixes (a typo, a missing import, a variable name), GitHub can generate the fix for you. Click the ± button on a line-level comment to open the suggestion editor:

```suggestion
def authenticate(user_id: int, token: str) -> User:
    """Authenticate a user by ID and token."""
    if not token:
        raise ValueError("Token is required")


The suggestion appears as a code block in the comment. The author can click **Commit suggestion** to apply it directly, or **Add to batch** to apply multiple suggestions at once. Suggested changes are powerful because they turn a review comment into a one-click fix — no back-and-forth about what the author meant to write.

### Review Summary

When you finish a review on GitHub, you choose one of three dispositions:

| Disposition | Meaning |
|---|---|
| **Comment** | General feedback, no formal approval or request for changes |
| **Approve** | The PR is ready to merge as-is |
| **Request changes** | The PR has blocking issues that must be addressed before merging |

After choosing a disposition, you write a **review summary** — a top-level comment that explains the overall feedback. This summary is especially important when the PR is complex or when there are multiple blocking comments. It gives the author a narrative to go with the detailed line comments.

## CLI Review with `gh`

You can conduct a full review workflow from the terminal using the `gh` GitHub CLI. This is especially useful for DevOps engineers who spend most of their time in a terminal and want to avoid context-switching to a browser.

### Finding PRs That Need Your Review

```bash

review-requested:@me filters to PRs where your review is explicitly requested. You can also use assignee:@me to find PRs you authored.

Submitting a Review

Approve a PR:

gh pr review 42 --approve -b "Clean implementation. The retry logic is well-
handled and the error messages are much clearer now. Left a few nits on
style, but nothing blocking. Ship it!"

Approved pull request #42 (fix-user-auth-logic)

Request changes:

gh pr review 42 --request-changes -b "The retry logic looks good overall, but
it still does not handle the case where ALL retry attempts fail. Currently
it will throw whatever exception _call_auth_service raises on the last
attempt, but it does not wrap it in a meaningful error. Please add a
custom exception and retry the error message to explain what happened."

Pull request #42 (fix-user-auth-logic) reviewed with requested changes.

Leave a comment without a verdict:

gh pr review 42 --comment -b "Consider extracting this retry logic into a
separate decorator. That would make it reusable across other service calls
without duplicating the loop. Not blocking, just a suggestion for a future
refactor."

Comment submitted on pull request #42 (fix-user-auth-logic)

Review Checklist

Before you submit a review (approve, request changes, or comment), work through this checklist. You do not need to manually verify every item — scan for the ones relevant to the PR in front of you.

Correctness

  • [ ] Does the logic do what the description says it does?
  • [ ] Are there edge cases that are not handled? (empty inputs, null values, large inputs, concurrent access)
  • [ ] Are errors handled explicitly, or do they propagate silently?
  • [ ] Are return values used correctly by the caller?

Tests

  • [ ] Does the PR add or update tests for the new behavior?
  • [ ] Do the tests cover the edge cases, not just the happy path?
  • [ ] Do the tests assert outcomes, not just that code runs without crashing?

Documentation

  • [ ] Are new public functions, classes, or methods documented?
  • [ ] Are complex logic blocks explained with comments?
  • [ ] Was the README or other documentation updated if behavior changed?

Security

  • [ ] Are user inputs validated before use?
  • [ ] Are secrets (API keys, tokens, passwords) loaded from environment variables or a secrets manager, not hardcoded?
  • [ ] Are database queries parameterized to prevent SQL injection?
  • [ ] Are file paths sanitized to prevent path traversal attacks?

Performance

  • [ ] Are there any obvious N+1 query patterns or unnecessary loops?
  • [ ] Are large data structures processed lazily or in chunks where appropriate?
  • [ ] Are expensive operations (network calls, file I/O) handled with timeouts?

Style and Consistency

  • [ ] Does the code follow the project's established conventions?
  • [ ] Are variable and function names descriptive enough to understand without a comment?
  • [ ] Is there dead code or commented-out blocks that should be removed?

Responding to Reviews

Getting a review is not the end of the process — it is the start of a back-and-forth. Here is how to handle it professionally.

Acknowledge Every Comment

Even if you disagree with a comment, acknowledge it. A quick "Got it, addressed in the latest push" or "I had considered that — see my reply above" closes the loop. GitHub marks conversations as resolved by default when the author replies; reviewers can also resolve conversations they opened.

Ask for Clarification Early

If a reviewer asks for something that is unclear, do not guess. Ask:

"Could you clarify what you mean by 'handle the error'? Do you want me to
log it, return an error code, or raise a custom exception? Happy to do
whatever you prefer once I understand the goal."

This is faster than implementing something and having the reviewer ask for a different approach.

Push Focused Fixes

When you address review comments, keep your fix commits focused:

git add src/auth/token.py
git commit -m "Address review: add descriptive exception when all retries fail"
git push origin feature/auth-improvements

Reference the review comment in your commit message or in a PR comment so the reviewer can quickly verify the fix:

"@reviewer Done — I added a `TokenRefreshError` exception that wraps the
underlying error and includes the number of retries attempted. Screenshot
of the new error message: [link]"

Resolve Conversations

After the reviewer verifies your fix and marks a conversation resolved, it is done. Do not reopen old resolved conversations unless you have genuinely found a new issue.

Re-Request Review When Ready

After pushing all fixes, re-request a review from the same reviewers:

gh pr review 42 --request-reviewers sarahdev

Or use the GitHub UI — look for the Re-request review button on the PR page. This sends a notification to the reviewer that their attention is needed again.

Exercise

In this exercise you will play both the reviewer and the author. You will find a PR that needs your review, leave feedback, request changes, and then — after the "author" fixes the issue — approve the PR.

Scenario: You are on the DevOps team at a startup. Your colleague alexdev has opened PR #38 to add a health check endpoint to your API. You have been asked to review it. When you look at the diff, you notice a problem: the endpoint does not handle the case where the database connection fails — it will crash with an unhandled exception instead of returning a healthy status.

Time needed: ~10 minutes

Setup

Ensure you are logged into gh and have a repository with at least one open PR to work with. If you are following along with a real repository, replace owner/repo and PR numbers with your actual values.

gh auth status

github.com
  ✓ Logged in to github.com as your-username
  ✓ GitHub CLI: gh 2.x.x
  ✓ git: git 2.x.x

Tasks

Task 1 — Find PRs awaiting your review

List open PRs where your review is requested:

gh pr list --search "review-requested:@me" --state open --json number,title,url

[
  {
    "number": 38,
    "title": "Add health check endpoint",
    "url": "https://github.com/alexdev/your-repo/pull/38"
  }
]

Task 2 — Examine the PR diff

Open the diff for PR #38:

gh pr diff 38

diff --git a/api/health.py b/api/health.py
new file mode 100644
index 0000000..123abc
--- /dev/null
+++ b/api/health.py
@@ -0,0 +1,14 @@
+"""Health check endpoint for load balancer probes."""
+from flask import Blueprint, jsonify
+
+health_bp = Blueprint("health", __name__)
+
+@health_bp.route("/health")
+def health_check():
+    """Return service health status."""
+    from api.db import check_connection  # expensive import at request time
+
+    db_ok = check_connection()
+    return jsonify({"status": "ok", "db": db_ok})

Notice the comment # expensive import at request time — that is a red flag. Also notice there is no error handling around check_connection().

Task 3 — Leave a blocking review comment

As a comment on the PR (not a full review yet, just a single comment to start the conversation):

gh pr comment 38 --body "The health check looks like a good start, but there
is no error handling around \`check_connection()\`. If the database is
unreachable, this will raise an unhandled exception and return a 500 error.
Instead, it should catch that exception and return a 200 with \`db: false\`
so the load balancer knows the service is \`ok\` but the database is
unhealthy. That is the correct behavior for a health check."

https://github.com/alexdev/your-repo/pull/38#issuecomment-XXXXXXX

Task 4 — Request changes formally

Now submit a formal review with the "request changes" disposition:

gh pr review 38 --request-changes -b "Thanks for adding this! The structure
looks good. Blocking issue: \`check_connection()\` is not wrapped in a
try/except. If the DB is down, the endpoint crashes instead of returning a
health status. Please catch exceptions and return \`db: false\` rather than
raising. Non-blocking: consider moving the \`from api.db import ...\` import
to the top of the file — importing at request time adds latency to every
health check call."

Pull request #38 (Add health check endpoint) reviewed with requested changes.

Task 5 — Simulate the author fixing the issue

Pretend the author pushed a fix. Create a local "fix" commit to simulate the updated code:

git checkout -b simulate-fix
# Simulate the fixed health check file
cat > api/health.py << 'EOF'
"""Health check endpoint for load balancer probes."""
from flask import Blueprint, jsonify

health_bp = Blueprint("health", __name__)

@health_bp.route("/health")
def health_check():
    """Return service health status."""
    try:
        db_ok = check_connection()
    except Exception:
        db_ok = False

    return jsonify({"status": "ok", "db": db_ok})
EOF
git add api/health.py
git commit -m "fix: handle database errors gracefully in health check"
# Note: we are not actually pushing this — it is a simulation

[simulate-fix abc123d] fix: handle database errors gracefully in health check
 1 file changed, 4 insertions(+), 5 deletions(-)

Task 6 — Approve the PR

After the simulated fix, submit an approval:

gh pr review 38 --approve -b "The try/except handles database errors correctly
now. The health check returns \`db: false\` instead of crashing — exactly
what the load balancer needs. Non-blocking note from before about the import
is a future improvement, not a merge blocker. Thanks for the quick fix!
LGTM, ship it."

Approved pull request #38 (Add health check endpoint)

Task 7 — Verify the final state

gh pr view 38 --json reviews --jq '.reviews[] | {state, body}'

{"state":"COMMENTED","body":"The health check looks like a good start..."}
{"state":"CHANGES_REQUESTED","body":"Thanks for adding this..."}
{"state":"APPROVED","body":"The try/except handles database errors..."}

You have completed the full review cycle: examined the diff, left blocking feedback, requested changes, verified the fix, and approved.

Key Takeaways

  • Code review is collaboration, not judgment. Treat it as a shared responsibility for code quality.
  • Blocking vs. non-blocking matters. Be explicit about what must change before merge and what is optional.
  • GitHub's review tools (line comments, suggestions, review summaries) are designed for precise, efficient feedback.
  • gh brings the full review workflow to the terminal. You rarely need to open a browser.
  • Responding to reviews well is a skill. Acknowledge feedback, ask for clarification, push focused fixes, and re-request review when ready.
  • A good reviewer checks correctness, tests, documentation, security, performance, and style — in that order of priority.
